━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. アプリケーション概要
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Stravaなどで作成・記録したGPXファイルに「ナビ用ターンポイント（曲がり角案内）」を
追加して強化する、または地図上をゼロからクリックして新規ルートを作成するための
ルート編集ツール。生成したGPXは姉妹アプリ「gpx-navi」（index.html／ナビ本体）で
読み込んでターンバイターンのナビゲーションに使う。

主な機能:
  - 既存GPX（ルートデータ／実走行データ）の読み込みとターンポイント自動検出
  - 実走行データに対するマップマッチング（道路への吸着）と間引き
  - 地図クリックによる新規ルート作成・延伸・分岐編集
  - ターンポイントの手動追加・名称編集・削除
  - 国土地理院APIによる標高補正（元データ標高との比較・選択）
  - 交差点名・POI名の自動取得によるターンポイント命名
  - 完成したGPXのダウンロードおよびクラウドへの保存（招待ユーザー限定。19章）
  - クラウド上に保存済みのルート一覧からの読み込み（招待ユーザー限定。19章）

認証: クラウド保存・クラウド読み込み機能のみ「招待制ログイン」が必要（19章）。
それ以外の機能（ローカルGPXの読み込み・編集・ローカル保存/ダウンロード）は、
従来通り無認証・無制限で誰でも利用できる。


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
2. 画面全体構成
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

アプリは大きく2つの画面状態を持つ。「編集モード」に入っているかどうかで
表示内容が完全に切り替わる（同一画面内でのタブ切替ではなく、状態遷移）。

  状態A: スタート画面（未編集）
  状態B: 編集画面（GPX読み込み後 or 新規ルート作成モード）

編集モードに入る条件（いずれか）:
  - ローカルファイルがアップロードされている
  - ネットワーク上のルートがダウンロードされ、内容がセットされている
  - 「新規ルートを作成する」が選択されている（＝新規ルートモード）

編集モードを抜ける条件:
  - 「編集を破棄して戻る」→確認ダイアログで「スタート画面に戻る」を選択
    （この場合、編集中のデータはすべて破棄される。GPXの保存操作とは独立している）


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
3. スタート画面仕様
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

タイトル「🚴 gpx-editor vX.Y.Z」とサブキャプションを表示。
編集モードでない場合のみ、2カラムのカード型レイアウトを表示する。

────────────────────────────────────────
3-1. 左カード：GPXファイルを読み込む
────────────────────────────────────────

  1. データ種別のラジオボタン（必須選択、デフォルトは先頭項目）
     - 「🗺️ ルートデータ（Stravaルート作成など）」
     - 「🏃 実走行データ（GPSで記録した走行ログ）」
     補足ヘルプ: 「実走行データはマップマッチング・間引きを自動実行します」

  2. ローカルファイルアップローダー
     - 拡張子 .gpx または .xml のみ許可（それ以外はエラー表示して処理を止める）
     - アップロード時、ファイル名＋データ種別フラグを組み合わせた識別キーを生成し、
       前回と異なるキーであれば編集用の内部状態を全てクリアしてから読み込む
       （同じファイルの再アップロードでは状態を保持する）
     - 読み込み後、自動的に編集モードへ遷移する

  3. 「🔍 ルートを選ぶ」ボタン → ネットワークから読み込むダイアログを開く（3-3参照）
     招待ユーザーとしてログインしていない場合、このボタンは無効化
     （グレーアウト）し、「招待ユーザー限定の機能です」等のツールチップを
     表示する（19章）

────────────────────────────────────────
3-2. 右カード：新規ルートを作成する
────────────────────────────────────────

  説明文と機能リストを表示:
    - 📍 地図クリックでルートを延伸
    - ⚓ アンカーポイントのドラッグでルート編集
    - 🔍 交差点ターンポイントを自動検出
    - ⛰️ 国土地理院による標高補正（日本国内）

  「🗺️ 新規ルートを作成する」ボタン押下で、編集用の内部状態を全てクリアし、
  新規ルートモードのフラグを立てて編集画面へ遷移する（この時点ではまだ
  ルート点は0件）。

────────────────────────────────────────
3-3. ネットワークから読み込むダイアログ
────────────────────────────────────────

  モーダルダイアログ「☁️ ネットワークから読み込む」。

  1. 開いた時点でSupabaseからルート一覧を取得する（1回のみ取得しキャッシュ、
     チェック操作のたびに再取得はしない）。
     - 取得失敗時: エラーメッセージを表示して終了
     - 0件の場合: 「ネットワーク上にルートがありません。」と表示して終了

  2. 一覧をテーブル表示する。各行はチェックボックス・ルート名・距離・獲得標高。
     - 距離: メートル値をkmに変換し小数1桁（例: "12.3 km"）。未取得は "---"
     - 獲得標高: メートル値を整数表示（例: "340 m"）。未取得は "---"
     - チェックボックスは単一選択のみ許可。複数チェックされた場合は最後に
       チェックされた1件のみ有効にする。全解除も可能（選択なし状態に戻る）

  3. 選択中のルートがあれば、名前・距離・獲得標高を強調表示する。
     選択がなければ「チェックボックスでルートを選択してください」と案内。

  4. ボタン行:
     - 「キャンセル」: ダイアログを閉じ、一覧キャッシュ・選択状態を破棄
     - 「読み込む →」: 選択がない場合は無効化。押下でGPXをダウンロードし、
       成功したら編集用の内部状態を全てクリアしてから、ダウンロードした内容を
       ロードして編集画面へ遷移する。失敗時はエラー表示してダイアログに留まる

────────────────────────────────────────
3-4. 招待ユーザー向けログイン導線
────────────────────────────────────────

  スタート画面に、招待ユーザーがログインするための導線（設定アイコン等）を
  常設する。押下すると、ログイン用の小さなパネル/ダイアログが開く:
    - 未ログイン時: ログイン方法を案内する入力欄と実行ボタンを表示する
      （具体的な認証方式は実装依存。19章参照）
    - ログイン中: ログイン中であることが分かる表示（メールアドレス等）と
      ログアウト操作を表示する

  ログイン状態はアプリ全体で共有され、3-1・16-4章のクラウド関連UIの
  有効/無効判定に使われる。

4. 編集画面レイアウト
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

編集モードに入ると、上から以下の順にセクションが並ぶ。

  1. 「↩ 編集を破棄して戻る」ボタン
     - 押下で確認ダイアログ「編集中の内容は破棄されます。スタート画面に戻りますか？」
       とボタン2つ「🏠 スタート画面に戻る」「✏️ 編集を続ける」を表示
       （具体的な表示方式・見た目はimplement.txt側の実装判断に委ねる）
     - 「スタート画面に戻る」で全内部状態を破棄しスタート画面へ

  2. 情報バナー（GPXにwptが既に含まれていた場合のみ）:
     「📂 ターンポイントを含んだGPXファイルを読み込みました。
       マップマッチング・標高補正はスキップされています。」

  3. ルート情報メトリクス:
     - ルート名（GPXのトラック名 → ファイル名 → "新規ルート" の優先順で決定。
       新規作成モードはトラック名・ファイル名とも無いため常に"新規ルート"になる）
     - GSI標高タグ「（GSI標高）」: 読み込んだGPXにeleSource=gsiフラグ
       （5-4章）が付いていた場合にのみ、ルート名の直後に表示する
       （フラグは読込時点のスナップショットであり、その後の編集で
       変化しても表示は更新しない）
     - 総距離（km、小数1桁）
     - 獲得標高（現在選択中の標高データ列を使って上りの合計を計算。m、整数。
       標高データが1点も無ければ "--- m"）

     表示形式:「{ルート名}{GSI標高タグ}　総距離 {X} km　獲得標高 {Y} m」

  4. メインエリア（2カラム、比率 2:1）
     - 左（col_map）: 地図プレビュー + 標高プロファイルグラフ（7章）
     - 右（col_list）: ターンポイント一覧パネル（15章）

  5. 区切り線

  6. 「💾 ルートを保存」ボタン（押下で保存画面へ遷移する。16章）


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
5. データモデル
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

アプリの中心データは「RoutePoint」の配列（route_points）。GPXのtrkpt 1点に
1対1対応し、配列の順序＝ルート上の走行順序。

────────────────────────────────────────
5-1. RoutePoint 構造
────────────────────────────────────────

  フィールド      型            説明
  ----------     ---------      --------------------------------
  lat            float          緯度
  lon            float          経度
  ele_org        float|null     元データの標高（GPS/GPXファイル由来、スパイク補正済み）
  ele_fix        float|null     国土地理院APIで取得した標高（スパイク補正済み）
  is_acpt        bool           アンカーポイントか（5-2参照）
  wpt            object|null    ターンポイント情報（5-3参照）。null=非ターン点
  changed        bool           直近のルート編集で新規生成された点か
                                 （ターンポイント再検出の対象範囲を決めるフラグ）

────────────────────────────────────────
5-2. acpt（アンカーポイント）
────────────────────────────────────────

  ルート編集の「つかみ手」となる点。地図上でドラッグ／削除の対象になる。
  常に以下を満たす:
    - 配列の先頭点と末尾点は必ず acpt（ルートが1点以上ある限り）
    - 中間点は、ユーザーがクリックでアンカーを挿入した箇所にのみ存在する
    - 連続するacpt区間（またはacptとwpt境界の間）が「1セグメント」であり、
      セグメント内の中間trkptはルーティングAPIの結果でしか生成されない
      （手動移動はできない）

────────────────────────────────────────
5-3. wpt（ターンポイント／ナビ案内点）
────────────────────────────────────────

  wpt構造: { name: string, delta: float|null }
    - name:  ナビゲーションで読み上げる案内文言（自由記述可、詳細は8章末尾）
    - delta: 検出時のベアリング変化量（度、符号付き）。手動追加やスタート/ゴールは null

  ルール:
    - 配列の先頭点・末尾点には必ず wpt が付く（name="スタート" / "目的地"、delta=null）
      ※ acptかつwptの両方を兼ねることができる（先頭・末尾は両方成立するのが通常）
    - is_acpt と wpt は独立したフラグであり、同一trkptが両方を持つことも
      片方だけを持つこともある
    - ターンポイントを削除すると、そのtrkptはwptを失うがacptとして残る
      （地図上のマーカー種別が変わるのみで、trkpt自体は消えない）

────────────────────────────────────────
5-4. GPXファイルとの対応（拡張タグ）
────────────────────────────────────────

  標準GPX 1.1のtrkpt/wptに加え、本アプリ独自の拡張名前空間
  "https://gpxnavi"（プレフィックス gpxnavi）を使用する。

  - trkpt/extensions/gpxnavi:acpt = "1"
      → そのtrkptが acpt であることを示す（読み込み時に復元。無ければ
        先頭・末尾のみをacpt扱いにフォールバック）

  - wpt/desc = "bearing_change:{delta:.1f}"
      → 自動検出されたターンポイントの delta 値を文字列で埋め込む
        （手動追加の場合は desc="manually added"）

  - wpt/name = ユーザーが確定した案内文言

  - trk/extensions/gpxnavi:eleSource = "gsi"
      → このファイルのtrkpt標高値が既に国土地理院データ（スパイク除去済み）
        由来であることを示す（16-3章でele_fixとして保存した場合にのみ
        付与する）。読み込み時にこのフラグがあれば、ele_orgと同じ値を
        ele_fixにもコピーする（16-2章）。org選択で保存した場合や、
        フラグが無い状態で保存した場合は付与しない（保存の都度、
        直前の状態を踏まえて上書きする）

  読み込んだGPXに既にwptが含まれている場合（_has_wpts）、そのGPXは
  「既にナビ情報を確定済み」とみなし、マップマッチング・ターン自動検出・
  標高補正は一切実行しない（6章参照）。

6. 読み込み時の自動処理パイプライン
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

GPXを読み込んだ直後（新規ルートモードを除く）、以下の順で自動処理が走る。
一度完了したら再読み込みされるまで繰り返さない。

  ステップ1: GPXパース
    - trkpt列を (lat, lon) のリストとして抽出
    - trkptが6点未満ならエラー表示して停止
    - waypointの有無を判定（_has_wpts）
    - gpxnavi:acpt 拡張タグからacptインデックス集合を復元

  ステップ2: マップマッチング（実走行データ かつ 未確定GPXの場合のみ。10章）
    - ルートデータ、または既にwpt確定済みのGPXの場合はスキップし、
      trkpt列をそのまま採用する

  ステップ3: route_points 初期化（マップマッチング完了後）
    - マッチング後の座標列（またはスキップ時は元のtrkpt列）からRoutePoint配列を生成
    - acpt復元: 拡張タグ由来の集合が2点以上あればそれを採用、なければ
      先頭・末尾のみをacptにする
    - 元データ標高処理: GPX内の標高値をスパイク除去（12章）にかけてele_orgへ格納
      （マップマッチングで間引かれている場合は間引き前後のインデックス対応を
      とって標高値を再マッピングする）
    - 国土地理院補正標高（ele_fix）の背景取得を、この初期化直後から開始する
      （13-2章参照。以降のルート編集操作でも継続的に追従する）
    - wpt初期化:
      - 既にwptを含むGPXの場合: 各waypointを最近傍trkptに割り当てる
        （直前に割り当てたインデックス以降から探索することで順序を保証）。
        desc の "bearing_change:" プレフィックスからdeltaを復元。
        先頭・末尾にwptが無ければ「スタート」「目的地」を補う
      - wptを含まないGPXの場合: ターン自動検出（11章）を実行し、検出された
        各ターンに対して交差点名取得（12章）を行い、名称付きwptを生成。
        先頭・末尾は無条件で「スタート」「目的地」に上書きする
    - 全点の changed フラグを false にする（初期化直後は「未変更」扱い）

  新規ルートモードの場合はステップ1〜3をスキップし、route_points=[] で開始する
  （最初の地図クリックで8-1の処理が走る）。

7. 地図の仕様
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

地図はOpenStreetMapタイルを使うインタラクティブ地図。画面幅は使い切り、縦方向は
画面の高さいっぱいに収まるよう、標高グラフ・ターンポイント一覧を含めて編集画面
全体が縦スクロールなしで収まることを優先し、地図の高さは残り領域に応じて可変とする
（2026-07-04改訂。以前は「500px程度の高さ」固定としていたが、画面サイズを
最大限活用する方針に変更した）。

────────────────────────────────────────
7-1. 描画レイヤー
────────────────────────────────────────

  - トラックライン: 全trkptを結んだポリライン
  - acptマーカー: 白丸に濃紺の枠線、ドラッグ可能、ツールチップ「acpt:N」
  - wptマーカー:
    - 先頭（スタート）: 緑ピン、ラベル"S"
    - 末尾（ゴール）: 赤ピン、ラベル"G"
    - 中間: turn_label（11章末尾）の色に応じた円マーカー
  - 標高グラフ連動カーソル: 標高グラフhover位置に対応するtrkpt上に矢印アイコン
    （進行方向へ回転）を表示（7-3参照）

  地図の中心・ズームは「地図クリック等の操作後は前回位置を維持」する仕様。
  実装上はブラウザのローカルストレージに中心・ズームを保存し、次回描画時に
  優先利用する。ルート読み込み直後や「地図移動」操作直後など、明示的に
  中心を合わせたい場面ではサーバー側指定の中心を強制的に使う
  （force_center相当のフラグ）。

────────────────────────────────────────
7-2. クリック時の候補選択
────────────────────────────────────────

  地図クリック位置から、画面上のピクセル距離にして一定の閾値（既定20px、
  ズームレベルに応じて実距離換算）以内にあるtrkptを全て候補として拾う。

  候補が0件 → 「空クリック」として扱う（ルート延伸へ、7-4参照）

  候補が1件以上ある場合、ルート距離にして200m以内の候補は1グループに
  クラスタリングし、各グループから最も近い1点を代表点とする。

  - 代表点が2グループ以上ある場合:
    ポップアップで候補一覧を表示する。既にwptが割り当たっている点は
    「✏ 「{name}」を編集 ({distance_km}km)」、割り当たっていない点は
    「＋ 新wptを追加 ({distance_km}km)」として選択肢を出す
  - 代表点が1グループのみの場合:
    最近傍trkptに対してアクションメニューを表示する:
      - 📍 ゴールを延長する
      - ⚓ アンカーポイントを挿入する
      - 🔀 ターンポイントを追加する
      - ✖ キャンセル

  トラックライン自体（trkptの間の線分）をクリックした場合も、最近傍trkptに
  対して同じアクションメニューを表示する。

  wptマーカーを直接クリックした場合:
    候補が1件のみならその点の編集（wpt_click）、複数候補があれば同様の
    候補一覧ポップアップを表示する。

────────────────────────────────────────
7-3. 標高グラフとの相互連携
────────────────────────────────────────

  - 標高グラフ上でカーソルをホバーすると、対応する走行距離(km)に最も近い
    trkptの位置に、進行方向を向いた矢印マーカーが地図上に表示される
    （ホバーが外れると消える）
  - 同時に、その地点の標高値（m、現在表示中の標高データ系列の値）を
    標高グラフ上にも表示する（ホバーが外れると消える）
  - 標高グラフをクリックすると、対応するtrkptに対して:
    - 既にwptが割り当たっていればそのwptの編集（wpt_click相当）
    - 割り当たっていなければ「⚓ アンカーポイントを挿入する」
      「🔀 ターンポイントを追加する」「✖ キャンセル」のアクションメニューを表示

────────────────────────────────────────
7-4. 地図イベントとサーバー側処理
────────────────────────────────────────

  地図から発生するイベント種別（8章の状態遷移で処理される）:

  - click_empty: 空白部クリック（緯度経度）
  - acpt_drag_end: acptマーカーのドラッグ終了（対象acptのインデックス、新座標）
  - acpt_delete: acptマーカーの右クリックで削除確認ポップアップ
    （「🗑 削除する」「✖ 何もしない」）を表示し、「削除する」が選択された
    場合にのみ発火する（コンテキストメニューは抑止する）
  - dialog_result(action=extend|acpt|wpt): クリックメニューでの選択
    （extend=ゴール延長、acpt=アンカー挿入、wpt=ターンポイント追加）
  - wpt_click: wptマーカー（または候補一覧経由）のクリック→一覧パネルへフォーカス
  - wpt_delete: wptマーカーの右クリックで削除確認ポップアップ
    （「🗑 削除する」「✖ 何もしない」）を表示し、「削除する」が選択された
    場合にのみ発火する（コンテキストメニューは抑止する）

  各イベントには常に発生時点の地図中心・ズーム・タイムスタンプが付随する。
  同一イベントの重複処理を避けるため、タイムスタンプで既処理判定を行う。


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
8. 地図イベントの処理仕様（状態遷移）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

以下のうち、既存ルートに変更を加える操作（8-1でacptが既に存在する場合の延伸、
8-2、8-3、8-4、8-5）は、直前の route_points 全体を「1回だけ戻せるUndoスナップ
ショット」として保存してから変更を適用する（14章）。ただし8-1で route が空の
状態から最初の1点を追加する場合（元に戻すと空ルートに逆戻りするだけで実用上の
価値が薄いため）と、8-6（wpt_click、地図移動のみで変更を伴わない）は対象外。

────────────────────────────────────────
8-1. click_empty / dialog_result(extend) — ゴール延長
────────────────────────────────────────

  - acptが1つも存在しない（＝ルートが空、新規ルートモードの最初のクリック）:
    クリック位置を新しい唯一の点として追加する。is_acpt=true、
    wpt={name:"スタート", delta:null}。ルーティング計算はしない
  - acptが既にある場合:
    現在の最終acpt座標からクリック位置までを外部ルーティングAPI（9章）で
    経路計算し、その区間点列（起点を除く）を末尾に追加する。
    - 追加された区間の末尾点を新しい is_acpt にする
    - 追加された区間の末尾点に wpt={name:"目的地", delta:null} を設定する
    - 旧・末尾点が wpt.name=="目的地" だった場合はそのwptを解除する
      （trkptとしては残る。「新規trkptの中間点」に格下げされるイメージ）
  - 処理後、route_modified フラグを true にする
    （＝ターンポイントの再検出を促す警告表示につながる。8-6参照）

────────────────────────────────────────
8-2. acpt_drag_end — アンカーポイントの移動
────────────────────────────────────────

  ドラッグされたacptの位置（先頭・末尾・中間）によって処理が変わる。

  - 先頭acptの移動:
    次の境界点（次のacptまたはwptのうち最小インデックス。無ければ末尾）までを
    新座標から再ルーティングし、その区間で先頭からその境界点手前までを置換する。
    新しい先頭点に wpt={name:"スタート", delta:null} を再設定する
  - 末尾acptの移動:
    前の境界点から新座標までを再ルーティングし、その区間で境界点の次から
    末尾までを置換する。新しい末尾点に wpt={name:"目的地", delta:null} を再設定する
  - 中間acptの移動:
    前の境界点から新座標まで、および新座標から次の境界点までをそれぞれ
    再ルーティングし、2つの区間を結合してその範囲を置換する。
    結合点（新座標に対応する点）を新しい is_acpt とする

  ここでいう「境界点」とは、動かした点を除いた直近のacptまたはwptのうち、
  対象より前方/後方にある最初の点を指す（5-2参照）。

────────────────────────────────────────
8-3. acpt_delete — アンカーポイントの削除
────────────────────────────────────────

  右クリックすると削除確認ポップアップが表示され、「🗑 削除する」を選んだ
  場合にのみ以下の処理が実行される。「✖ 何もしない」を選ぶ、またはポップ
  アップ外をクリックして閉じた場合は何も変更しない。

  - acptが1つしかない場合: ルート全体を空にする
  - 先頭acptの削除: 次のacptまでのtrkptを全て切り捨てる。新しい先頭に
    is_acpt=true を付け、wptが無いか名前が「スタート」でなければ
    {name:"スタート", delta:null} を設定する
  - 末尾acptの削除: 前のacptまでのtrkptを全て切り捨てる。新しい末尾に
    is_acpt=true を付け、wptが無いか名前が「目的地」でなければ
    {name:"目的地", delta:null} を設定する
  - 中間acptの削除: 前後の境界点間を再ルーティングし、その区間（両端は含まない）
    を置換する。削除したacptがwptを兼ねていた場合はそのwptも失われる

────────────────────────────────────────
8-4. dialog_result(action=acpt) — アンカーポイントの挿入
────────────────────────────────────────

  クリックで選ばれた最近傍trkpt位置に新たなacptを挿入する。
  前後の境界点それぞれとの間を再ルーティングし、2区間を結合して
  該当範囲を置換する。挿入点を is_acpt=true にする。

────────────────────────────────────────
8-5. dialog_result(action=wpt) — ターンポイントの追加
────────────────────────────────────────

  クリックで選ばれた最近傍trkptに既にwptが付いていなければ、以下の順で
  名称候補を決定する:

  1. 前後trkptとのベアリング差（delta）が計算できる中間点の場合:
     交差点名取得（12章、半径20m・タイムアウト5秒・1URLのみ試行）を行い、
     見つかれば「{交差点名}を{方向ラベル}」、見つからなければ方向ラベルのみ
     （turn_label、11章末尾参照）
  2. 先頭・末尾で delta が計算できない場合:
     交差点名が見つかればそれをそのまま名前にする
  3. 交差点名も見つからない場合:
     POI名取得（12章フォールバック）を試み、見つかれば「「{POI名}」」、
     見つからなければ「追加したターンポイント」

  is_acpt は変更しない（ターンポイント追加はacptの有無と独立）。

────────────────────────────────────────
8-6. wpt_click — 一覧パネルへのフォーカス
────────────────────────────────────────

  クリックされたwptの座標を地図の新しい中心にし、右側の一覧パネル内で
  対応する名前入力欄までスクロールしてフォーカスする（16章）。

────────────────────────────────────────
8-7. wpt_delete — ターンポイントの削除（地図上の右クリック）
────────────────────────────────────────

  wptマーカーを右クリックすると削除確認ポップアップが表示され、
  「🗑 削除する」を選んだ場合にのみ以下の処理が実行される。
  「✖ 何もしない」を選ぶ、またはポップアップ外をクリックして閉じた
  場合は何も変更しない。

  15章の一覧パネルの🗑削除ボタンと同じ処理を行う: 対象trkptのwptを
  外し、is_acpt=trueにする（ただのアンカーポイントとして残る。
  8-3のacpt_delete（ルート区間そのものを短縮する処理）とは異なる）。
  スタート・ゴールのwptも一覧パネルと同様に区別なく削除できる。


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
9. ルート延伸・再計算ロジック（ルーティングAPI利用）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

8章の各操作で「区間を再計算する」際は、外部の道路ルーティングAPI（17章）に
2点以上の緯度経度を渡し、道路に沿ったtrkpt列を得る。

  - 自転車（bicycle）モードで計算する
  - APIが失敗した場合は、渡した点をそのまま直線で結ぶ（フォールバック）
  - 返却された点列で該当区間のtrkptを完全に置換する（点数が変わってよい）
  - 新規生成された中間点は changed=true とする（ターンポイント再検出の対象になる）


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
10. マップマッチング（実走行データ専用）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

実走行データかつwpt未確定のGPXを読み込んだ場合にのみ自動実行する。
ルートデータの場合は完全にスキップする（trkptをそのまま使う）。

────────────────────────────────────────
10-1. 前処理（間引き）
────────────────────────────────────────

  読み込んだtrkpt全点に対し、Douglas-Peucker（RDP）法で間引きを行う
  （許容誤差 約5m相当）。間引き後に残ったインデックス列を記録しておき、
  後段の標高値マッピング（6章ステップ3）で使用する。

────────────────────────────────────────
10-2. チャンク処理と進捗表示
────────────────────────────────────────

  間引き後の点列を50点単位のチャンクに分割し、チャンクごとに
  マップマッチングAPI（自転車専用、探索半径50m）へ逐次リクエストする。

  進捗ダイアログ「🗺️ マップマッチング中」を表示し、以下を提供する:
    - プログレスバー（現在チャンク数／全チャンク数）
    - 「⏹ キャンセル」ボタン

  マッチング結果のうち type が "matched" または "interpolated" の点のみを
  座標置換の対象とする（それ以外はRDP間引き後の元座標を保持）。

  終了条件:
    - 全チャンク処理完了 → ステータス「完了」（1点以上置換されていれば）
      または「エラー」（0点）
    - ユーザーがキャンセル → ステータス「キャンセル」。それまでに処理済みの
      部分結果を採用する
    - 1チャンク目でタイムアウト → 自動的に全体をキャンセル扱いにし、
      間引き後の元座標をそのまま採用する（"1チャンク目タイムアウトにより
      自動キャンセル"）
    - 2チャンク目以降のタイムアウトやエラーは、そのチャンクだけ元座標を
      保持しつつ処理を継続する（エラーメッセージを蓄積）


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
11. ターン検出アルゴリズム
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

「角度法」によるコーナー検出。

  パラメータ既定値:
    - min_turn_angle = 45度（これ未満は直進とみなし候補にしない）
    - min_dist = 100m（近接する候補の重複排除距離）
    - smooth = 1（前後何点をベアリング計算に使うか）

  アルゴリズム:
    1. 各点X（配列の smooth 番目から末尾-smooth番目まで）について、
       smooth点前のA→Xのベアリングと、XからXのsmooth点後のBへのベアリングを求める
    2. 2つのベアリングの符号付き差分（-180〜180度、正=右旋回・負=左旋回）を
       「turn」とする
    3. |turn| >= min_turn_angle の点を候補とする
    4. 候補を|turn|の大きい順に処理し、既に採用候補から min_dist 未満の
       距離にある他の候補は棄却する（同一コーナーの重複検出を防ぐ）
    5. 採用された候補を元のインデックス順に並べ替えて返す

  用途:
    - GPX初回読み込み時、wptが含まれていなければ全区間に対して実行
    - 「🔍 ターンポイント検出」ボタン押下時、直近の編集で changed=true に
      なっている連続区間ごとに実行（既にwptがある点は上書きしない）

  方向ラベル（turn_label）は delta の値に応じて以下に分類される:
    - delta >= 60   : 右折　⇒　赤系
    - delta >= 25   : やや右　↗　オレンジ系
    - delta <= -60  : 左折　⇐　青系
    - delta <= -25  : やや左　↖　紫系
    - それ以外       : 直進維持　↑　グレー系
  （手動追加でdeltaがnullの場合は 📍 緑系のアイコンを使う）


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
12. 交差点名・POI名取得
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

外部の地図データベース検索API（Overpass、17章）を用いて、ターンポイント
付近の交差点名やランドマーク名を取得し、wptの命名に利用する。

────────────────────────────────────────
12-1. 交差点名検索
────────────────────────────────────────

  各ターン候補点について、半径20m以内にある
  「信号交差点／横断歩道／一時停止標識／ミニロータリー／幹線分岐点」
  タグかつnameタグを持つノードを検索する。複数のターン候補を1回の
  クエリでまとめて検索する（バッチ）。

  最近傍ノードとの距離が半径以内であれば、そのノードのname値を採用する。
  取得できなかった点はキーなし（命名候補なし）として扱う。

  検索先は複数のミラーサーバーを優先順に用意し、先頭から順に試して
  最初に成功したレスポンスを採用する（フェイルオーバー）。

────────────────────────────────────────
12-2. POI名検索（フォールバック）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  交差点名が見つからなかった単一地点について、観光・商業・レジャー・史跡・
  自然・店舗タグのいずれかを持つ最寄りのnameタグ付きノードを検索する
  （8-5のケース3で使用）。


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
13. 標高処理
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

各RoutePointは「元データ標高（ele_org）」と「国土地理院補正標高（ele_fix）」の
2種類の標高値を独立に持ち、出力時にどちらか一方を選んで使う。

────────────────────────────────────────
13-1. 元データ標高
────────────────────────────────────────

  読み込んだGPXにelevation値が含まれていれば、スパイク除去（13-3）を通した
  上でele_orgに格納する。含まれていなければele_orgは全点null。

────────────────────────────────────────
13-2. 国土地理院補正標高（リアルタイム背景取得＋補正）
────────────────────────────────────────

  国土地理院補正標高（ele_fix）は、ユーザーの明示操作を待たず、ルートの
  作成・編集（新規点の追加、区間の再ルーティング、アンカー移動など）に
  追従して自動的に背景取得される。

  対象範囲: 日本国内（緯度24〜46度、経度122〜154度の矩形内）にある
  RoutePointのみ。ルートの先頭点が国外にある場合、この機能自体を
  無効化する（画面上には「対象外」である旨を表示する）。

  取得タイミング・範囲:
    - 取得対象は ele_org（元データ標高）が無い（null）点に限る。GPXに
      元々含まれていた標高値がある点は、既に信頼できるデータがある
      とみなし、国土地理院への問い合わせを行わない（無駄なAPI呼び出し
      を避けるため）。新規ルート作成モードなど、元データ標高が全く
      無いルートでは全点が対象になる
    - route_pointsに新規点が追加された、または既存点の座標が変わった
      場合、その変更点（および変更によって新たに生成された中間点）の
      うちele_orgが無いものについてのみ ele_fix を取得する。既に
      取得済みで座標も変わっていない点は再取得しない
    - ルート編集操作が連続して発生する間（ドラッグ中など）は取得を
      すぐには開始せず、操作が一段落してから開始する（無駄なリクエスト
      の発生を避けるため）
    - 取得は非ブロッキングで行われ、進行中も地図・一覧・GPX出力操作を
      妨げない。取得の進行状況は控えめなインジケータ（例:
      「⛰️ 標高取得中… N/M点」）で示す程度とし、モーダルで画面全体を
      塞がない
    - スパイク除去（13-3章）と勾配統計の再計算は、取得の途中経過ごとに
      逐次行わない。全RoutePointについてele_fixの取得試行が完了（成功、
      または再試行を尽くして最終的に取得不可と確定）した時点で、
      まとめて1回だけ実行する。理由: スパイク除去は前後の点（最大600m
      先までのアンカー探索、150m幅の中央値ウィンドウ）を参照するため、
      取得途中の欠けたデータに対して部分実行すると境界付近で誤補正が
      発生し、後続の取得で結果が変わってしまう（チラつき・手戻り）
    - 取得不可のまま残った点がある場合でも、上記のタイミングで1回だけ
      実行する（一部nullを含んだ状態で確定させる。全点埋まっていない
      限りfix側は「完全なデータセット」として扱われない、という
      13-4章の判定はそのまま維持する）

  取得失敗時の扱い:
    - 個別点の取得に失敗した場合は自動的に再試行する
    - 再試行を重ねても取得できない点は「取得不可」（null）のまま残し、
      その旨を控えめに表示する（route全体の編集は妨げない）
    - 標高値が -9999 または "-----" の場合も「取得不可」としてnull扱い
      にする
    - ユーザーが手動で「取得できなかった点だけ再試行する」操作を
      呼び出せるようにする（明示的なやり直し手段は残す）

  この背景取得はあくまで編集中の標高グラフ表示を追従させるための
  ものであり、GPXとして保存する際にどちらの標高系列を使うかの確定、
  および保存に必要な追加取得は行わない。保存直前の整合性チェックと
  それに伴う再取得は16章（保存画面）で別途行う。

────────────────────────────────────────
13-3. スパイク除去アルゴリズム
────────────────────────────────────────

  標高API/GPS由来の局所的な異常値（スパイク）を検出し、前後の安定区間を
  結ぶ直線補間で置き換える。

  概念:
    1. 隣接点間の勾配（%）を全区間で計算する
    2. 勾配の絶対値が閾値（既定15%）以上、かつ標高差が一定値（既定2m、
       ただしセグメント長10m未満は距離条件を免除）以上の区間を
       「異常区間」候補とする。より厳しい閾値（35%・3m）を超える区間も
       同様に候補に加える
    3. 異常区間の中心から一定距離（既定250m）以内にある、やや緩い基準
       （勾配15%以上）の区間も「近傍」として巻き込む
    4. 近傍区間同士が近接していれば1つの「クラスタ」にまとめる
    5. 各クラスタについて、その前後で「安定した基準点（アンカー）」となる
       点を探索する。アンカー候補の条件:
       - 前後の勾配がともに一定以下（既定12%）
       - 周農150m以内の標高中央値との乖離が一定以下（既定5m）
       - 2点以上連続して条件を満たすことを要求（既定600m以内で探索）
       クラスタの境界点自体がアンカー条件を満たす場合はそれを優先する
    6. 前後アンカー間の正味勾配が一定以下（既定15%）であれば、その区間の
       内側の全点をアンカー間の直線補間値に置き換える。ただし補正量が
       隣接区間長に対して一定割合（既定1%）未満の場合は据え置く
       （無意味な微小補正を避ける）
    7. 近接するクラスタ同士（既定50m以内）は補正前にマージする

  戻り値として、補正後の標高列に加え「補正クラスタ数」「補正点数」
  「補正前後の最大勾配絶対値」を統計情報として返す。

────────────────────────────────────────
13-4. 勾配統計と標高データの自動選択
────────────────────────────────────────

  各標高データ列（org / fix）について、上り最大勾配（正の最大値）と
  下り最大勾配（負の最小値）を算出する。

  デフォルトの選択ロジック（GPX出力・表示用ラジオの初期値）:
    - org・fix 両方が全点揃っている場合: (上り最大 + |下り最大|) の合計が
      小さい方（＝勾配の暴れが少ない方）を推奨として選ぶ。合計値が同値
      の場合はfixを優先する（GSIフラグを維持し、無駄な再取得を避けるため）
    - 片方のみ揃っている場合: 揃っている方を採用
    - どちらも揃っていない場合: org を既定にする（値は全点null）

  ユーザーはラジオボタンでいつでも表示を切り替えられる。fixが未確定
  （標高補正未実施）の状態でfixが選択されようとした場合はorgに強制する。

  ※ この選択ロジック・選択ラジオが実際にユーザーへ提示されるのは
  16-2章の保存画面のみである。編集画面（4章）にはこの選択UIを表示
  しない。ルートが編集済み（ele_orgが全点揃っていない）の場合は
  そもそも選択自体を提示せず、常にfixへ統一する（16-2章）。


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
14. Undo機能
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

以下の操作の直前に、route_points全体のディープコピーを1スロットだけ保存する。
「↩ 戻す」ボタンは直前の1回分のみ復元可能（スタック無し、上書き方式）。

  Undo対象に含む操作:
    - 8-1（acptが既に存在する状態でのゴール延伸）
    - 8-2（acptの移動）
    - 8-3（acptの削除）
    - 8-4（アンカーポイントの挿入）
    - 8-5（ターンポイントの追加）
    - 15章「🔍 ターンポイント検出」ボタンによる一括検出・命名
    - 15章「🗑」ボタンによるターンポイントの削除

  Undo対象に含まない操作（対象を広げても実用上の価値が薄い、または
  そもそもroute_pointsを変更しないもの）:
    - 8-1でルートが空の状態から最初の1点を追加する操作
      （元に戻すと空ルートに逆戻りするだけのため）
    - 8-6（wpt_click。地図移動とフォーカス移動のみで変更を伴わない）
    - 15章の名前入力欄への直接編集（連続的なテキスト入力であり、単一の
      スナップショットに馴染まないため。ブラウザのテキストフィールド
      ネイティブのUndo機能に委ねる）
    - 標高データ選択ラジオ（org/fix）の切り替え（表示切替であり
      route_pointsを変更しないため）

  復元後、復元した内容にwptが含まれる点が1つでもchanged=trueであれば
  「ルートが変更されています」警告状態を再度有効にする。


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
15. ターンポイント一覧パネル（右カラム）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  見出し「📋 ターンポイント一覧　(N件)」

  操作行:
    - 「🔍 ターンポイント検出」: 実行前に14章のUndoスナップショットを保存
      した上で、changed=trueの点が1つでもあれば、その連続区間ごとに11章の
      検出＋12章の命名を実行し、まだwptが無い点にのみ新規wptを設定する
      （既存wptは上書きしない）。実行後、全点の changed を falseに戻し、
      route_modified を false にする
    - 「↩ 戻す」: 14章のUndoを実行（スナップショットが無ければ無効化）

  警告表示: route_modified が true かつ 1件以上wptが存在する場合、
  「ルートが変更されています。ターンポイント検出を実行してください。」

  一覧本体（スクロール可能、高さ固定）:
    各wpt付きtrkptについて1行:
      - 「{連番} | {方向矢印} | trkpt: {インデックス}」ボタン
        → 押下で地図中心をその点に移動し、名前入力欄にフォーカスを移す
      - 名前編集用テキスト入力（ラベル非表示、直接編集可能）
        入力値は即座にwpt.nameに反映される
      - 🗑 削除ボタン → 実行前に14章のUndoスナップショットを保存した上で、
        そのwptを解除する（is_acptはtrueにする。つまり削除してもtrkpt
        自体とアンカーとしての機能は失わない）

  一覧が空の場合: 「ターンポイントがありません。」

  末尾の使い方ヒント:
    「💡 地図をクリックして新しいポイントを追加。ナビゲーションの内容は、
    「左折」「やや左」「直進」「やや右」「右折」を推奨しますが、フリーワードです。
    「左」、「右」の文字を入れておくと逆走時に正しく変換されます。
    鍵カッコ「」で全体を括るとスポット名として解釈します。
    漢字名称の読み方を括弧（）で括ることで指定できます。」
    ※ これらの記法（逆走時変換・スポット名・読み仮名指定）は本アプリでは
      解釈せず、姉妹アプリ（ナビ本体）側での解釈規約である。本アプリは
      自由文字列としてそのままGPXのwpt nameに書き出すのみ。


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
16. GPX出力仕様
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

────────────────────────────────────────
16-1. 保存ボタンと保存画面
────────────────────────────────────────

  編集画面（4章）には「💾 ルートを保存」ボタンのみを置く
  （ターンポイントが1件も無い場合は無効化）。押下で保存画面（モーダル）
  を開く。標高データの選択UIや詳細なステータス表示は編集画面には置かず、
  全て保存画面（16-2章・16-4章）に集約する。

  なお、13-2章のリアルタイム背景取得の進行状況（控えめなインジケータ、
  例:「⛰️ 標高取得中… N/M点」）は編集中の参考表示として編集画面側に
  残してよいが、これは途中経過の目安に過ぎず、保存内容を確定するもの
  ではない。

────────────────────────────────────────
16-2. 標高整合性チェック（保存画面起動時）
────────────────────────────────────────

  保存画面を開いた直後、実際にGPXへ書き出す前に、標高データの整合性を
  確認・補完するステップを行う。ele_orgとele_fixは出典が異なり、絶対値に
  ズレが生じ得るため、出力するGPX内で2つの系列を点ごとに混在させる
  ことは許容しない。ルート全体を必ずどちらか一方の系列に統一してから
  出力する。

  取得対象: ele_fixがまだ確定していない（null）点のみを国土地理院から
  取得する（進捗表示付き。13-2章の背景取得とは独立に、ここで確実に
  対象点ぶんの取得を完了させる）。既にele_fixが確定している点は
  再取得しない。5-4章のeleSource=gsiフラグにより読込時点でele_orgから
  ele_fixへ複製済みの点はここに含まれるため、無駄な国土地理院API呼び出し
  を避けられる。

  取得完了後、ルート全体のele_fixに対して13-3章のスパイク除去を行う。

  判定はシンプルに、org・fixそれぞれのデータが「全RoutePointに揃って
  いるか」だけで行う（eleSource=gsiフラグや、ルートが編集されたか否かを
  個別に判定する特別処理は行わない）。

  - org_available = 全RoutePointにele_orgがある
  - fix_available = 取得完了後、全RoutePointにele_fixがある

  常に同じUIを表示する: ラジオボタンで元データ・国土地理院補正のいずれかを
  選ばせ、org_availableがfalseなら元データ側を、fix_availableがfalseなら
  国土地理院補正側を無効化する（13-4章の推奨ロジックは両方availableな
  場合にのみ適用し、「★推奨」を付与）。無効化の理由（編集済み、
  データ出典の混在等）を個別に説明する文言は表示しない。

  取得中はプログレス表示を行い、保存画面の以降のステップ（16-4章の
  ファイル名入力等）はこの整合性チェックの完了後に表示する。

────────────────────────────────────────
16-3. GPXビルド処理
────────────────────────────────────────

  GPX生成自体は以下の規則で行う:

  1. 読み込み元GPXがあればそれをベースにパースし、無ければ空のGPXを新規生成
  2. trkptは完全に route_points の内容で置き換える（点数が変わっていてもよい）。
     16-2章の整合性チェックで確定した標高列（org、またはfixに統一済みの値）
     があれば elevation として付与する
  3. トラック名: ベースGPXに名前が無ければ、保存画面で指定した
     ファイル名を採用する
  4. waypointは完全に再構築する。wpt.nameが空でdeltaがあればturn_labelの
     方向ラベルを使う。descには "bearing_change:{delta}" または
     "manually added" を格納する
  5. is_acptがtrueの点について、trkpt/extensions/gpxnavi:acpt="1" を
     XML後処理で追加する（gpxpyの標準出力にない拡張のため、生成されたXML
     文字列をパースし直して手動で挿入する。失敗時はextension無しで返す）
  6. eleChoice="fix"で保存する場合のみ、trk/extensions/gpxnavi:eleSource=
     "gsi"（5-4章）を付与する。それ以外（org選択、または既存フラグ）は
     常に一旦取り除いてから判定するため、古い状態を引き継がない

────────────────────────────────────────
16-4. 保存画面の内容
────────────────────────────────────────

  16-2章の標高整合性チェックが完了した後に、以下を表示する。

  1. ステータス表示:
     - ターンポイント: route_modifiedがtrueなら「⚠️ 未確定（ルート変更後に
       再検出を推奨）」、falseなら「✅ 設定済み（N個）」
     - 標高: 16-2章のチェックの結果（org/fix選択済み、またはfixに統一済み）
       を表示する
     - 上記の組み合わせにより、複合的な警告文を出し分ける
       （ターンポイントが未確定なパターンを含む）

  2. ファイル名入力（デフォルトは読み込み元ファイル名から拡張子・接尾辞を
     除いたもの、または "new_route"）。保存ファイル名は必ず
     "{入力値}_gne.gpx" になる（"gne" = GPX Navi Enhanced 相当の固定サフィックス）

  3. 「☁️ ネットワーク上にも保存」チェックボックス（直近の選択状態を記憶する）
     - 招待ユーザーとしてログインしていない場合、このチェックボックスは
       常に無効化（グレーアウト）され、オンにできない（「招待ユーザー限定の
       機能です」等のツールチップを表示する。19章）
     - ファイル名がASCII文字のみでない場合、Supabase保存用に別途
       半角英数字のファイル名入力を要求する（Supabase側のファイルキーは
       ASCII前提のため）。この入力が空の間はダウンロードボタンを無効化する

  4. ボタン行:
     - 「キャンセル」: 保存画面を閉じるのみ
     - 「⬇️ ダウンロード」: GPXファイルをブラウザ経由でダウンロードさせる。
       同時に「ネットワーク上にも保存」がオンならSupabaseへのアップロード
       （17章）を実行し、成功・失敗をトースト表示する
       （アップロード失敗時は保存画面を閉じずに留まる）

17. 外部サービス連携仕様
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

────────────────────────────────────────
17-1. 道路ルーティングAPI（区間再計算・9章）
────────────────────────────────────────

  用途: 2点以上の緯度経度から自転車向け道路ルートの座標列を取得する
  入出力: 緯度経度の配列 → 道路に沿った緯度経度の配列
  移動手段: 自転車固定
  フォールバック: 失敗時は入力点をそのまま結ぶ直線を返す
  タイムアウト目安: 30秒

────────────────────────────────────────
17-2. マップマッチングAPI（実走行データの道路吸着・10章）
────────────────────────────────────────

  用途: GPSトラック点列を最寄りの道路上にスナップする
  入力: 緯度経度の配列（1リクエストあたり50点以下）、探索半径50m、
        移動手段=自転車固定
  出力: 各点ごとのスナップ後座標と種別（matched/interpolated/unmatched等）。
        matched/interpolated のみ採用する
  タイムアウト目安: 1リクエスト30秒

────────────────────────────────────────
17-3. 標高API（国土地理院補正・13-2章）
────────────────────────────────────────

  用途: 単一地点の標高値（メートル）を取得する
  入力: 緯度・経度
  出力: 標高値（数値）。取得不可の場合は特殊値（-9999や"-----"相当）を返す
  並列度・バッチ方針: 13-2章の通りリアルタイム背景取得を行うため、具体的な
  並列数・バッチサイズは実装依存とする（implement.txt側で定義する）
  タイムアウト目安: 1リクエスト10秒。失敗点は再試行する

────────────────────────────────────────
17-4. 地図データベース検索API（交差点名・POI名・12章）
────────────────────────────────────────

  用途: 指定座標周辺のタグ付きノード（交差点・信号・POIなど）を検索する
  入力: クエリ言語で「半径・タグ条件」を指定するリクエスト
        （複数地点をまとめて1リクエストで検索可能）
  出力: 座標・タグ（nameタグ含む）を持つノードのリスト
  フェイルオーバー: 複数のミラーサーバーを優先順に用意し、先頭から順に
  試して最初に成功したものを採用する
  タイムアウト目安: 用途により5〜30秒（バッチ検索は長め、単発は短め）

────────────────────────────────────────
17-5. クラウドストレージ・データベース（Supabase・16-4章）
────────────────────────────────────────

  用途: 完成したGPXファイルの保存・一覧・ダウンロード

  ファイルストレージ:
    バケット: GPXファイル専用バケット
    ファイル名規則: "{ファイルキー}.gpx"
    アップロード時、同名ファイルが既に存在する場合はエラー（重複拒否）

  メタデータテーブル（1レコード=1ルート）:
    カラム         型          説明
    ------------   --------    --------------------------------
    file_key       text        ストレージ上のファイル名（拡張子なし相当）
    display_name   text        UI表示用のルート名（一意制約あり）
    distance_m     float       総距離（メートル）
    elevation_gain_m  float    獲得標高（メートル）
    created_at     timestamp   登録日時

  操作:
    - アップロード: ストレージへファイル保存後、メタデータをINSERT。
      メタデータ登録に失敗した場合はストレージ側もロールバック（削除）する。
      ファイルキー重複・表示名重複はそれぞれ専用のエラーメッセージを返す
    - 一覧取得: 登録日時の新しい順で全件取得
    - ダウンロード: file_keyを指定してストレージから取得しテキストとして返す

18. バージョン管理ルール
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  アプリケーションバージョン（例: 3.8.0）はセマンティックバージョニングに
  準じ、画面タイトル右肩に小さく表示する。

    - パッチ（x.y.Z）: バグ修正・細かい調整
    - マイナー（x.Y.0）: 新機能追加
    - メジャー（X.0.0）: 大規模リファクタ・破壊的変更

  実装移植時も、移植先での変更に対して同じ粒度でバージョンを更新する
  運用を維持する（実装仕様書（implement.txt）側で管理方法を定義する）。

19. 招待制クラウド機能
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

クラウド保存（16-4章）・クラウド読み込み（3-3章）は、あらかじめ運営者が
招待した特定ユーザーのみが利用できる機能とする。それ以外の一般ユーザーは、
ログイン不要でアプリの他の全機能（ローカルGPXの読み込み・編集・
ローカル保存/ダウンロード）を制限なく利用できる。

  - 招待: 運営者が個別にユーザーを登録する。アプリ内に一般ユーザー向けの
    自己サインアップ機能は設けない
  - ログイン導線: 3-4章の通り、スタート画面の常設アイコンから行う
  - 未ログイン時のクラウド機能: ボタン・チェックボックスを無効化
    （グレーアウト）し、ツールチップ等で「招待ユーザー限定の機能です」と
    案内する。機能の存在自体は一般ユーザーにも見える（非表示にはしない）
  - ログインで解禁される範囲: クラウド保存・クラウド読み込みのみ
    （他の機能への影響はない）
  - 具体的な認証方式（メールリンク等）・バックエンド側のアクセス制御方法は
    実装依存とする（implement.txt側で定義する）
